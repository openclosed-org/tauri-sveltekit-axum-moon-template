# AGENTS.md — 仓库级 AI 协作协议

## 1. 作用域与优先级

1. 本文件定义**仓库级、长期稳定**的协作规则。
2. 更近规则优先：子目录 `AGENTS.md`、`CLAUDE.md`、README、模块文档、测试规范 > 本文件。
3. 易过期内容（版本坑、调试案例、兼容性结论、命令模板）沉淀到 `.agents/playbooks/`，不要堆在本文件。

---

## 2. 默认目标

1. 以**最小改动、最小认知成本、可验证结果**完成任务。
2. 默认适用于：**独立开发者 + AI agent 协作维护**的中小型跨端项目。
3. 技术决策默认优先级：
   - 满足需求与验收标准
   - 正确性与边界条件
   - 回归风险与改动范围
   - 复用现有模式与项目约定
   - 可测试性与可维护性
   - 交付速度
   - 扩展性与性能优化

---

## 3. 全局硬约束

1. 默认使用**中文**沟通；代码、命令、配置键、日志、报错、协议字段保持原文。
2. **先读再改**：实施前必须先阅读相关代码、配置、文档、测试，禁止未审查现状就重写。
3. **先证据后判断**：遇到报错，先获取完整日志、复现步骤、版本信息，再分析和修改。
4. **先搜索后猜测**：涉及新框架、新版本、陌生 API、不确定语法时，优先使用 MCP / tools / skills / 官方文档 / GitHub Issues / Release Notes / webfetch 或 Web 搜索确认；禁止基于记忆连续盲改。
5. **先小后大**：优先最小闭环、局部化改动、可回滚方案，避免一次性铺大架构。
6. 修改现有实现前，先解释当前设计可能在保护什么；未经确认，不要随意改变可观察行为、接口形状、错误语义、时序和默认值。
7. 未实际执行的操作，不得声称已执行；未实际验证的结果，不得声称已验证。
8. 对任何不确定信息，必须明确说明不确定点、影响范围与后续验证方式。
9. 禁止回避问题、掩盖问题、制造“表面通过”。

---

## 4. 工程认知偏置

1. 默认这是 **Tauri 2 + SvelteKit + Axum** 跨端代码库；隐性约束与显性需求同等重要。
2. 实施前优先外显：
   - 隐性约束
   - 不变量
   - 契约边界
   - 失败模式
   - 回滚路径
3. 默认优化目标：
   - 概念一致性
   - 局部可推理
   - 信息隐藏
   - 契约优先
   - 显式不变量
   - 小爆炸半径
   - 可逆改动
   - 确定性与幂等性
   - 未来 agent 可持续编辑
4. 默认架构偏置：
   - adapter 薄，core 稳
   - Tauri command / Axum handler 主要承担适配与协议职责
   - 核心业务逻辑进入稳定、可测试的 core
   - SvelteKit 负责 UI 组合与瞬时状态，不承载领域真相
   - 跨 Rust / TypeScript 边界优先 typed/shared contracts
   - 优先 vertical slice 和局部修改
   - 宁可小范围重复，也不要过早抽象
5. 禁止过度设计；先做最小可验证闭环。

---

## 5. 工具与检索原则

1. 当 tools / skills / MCP / 子 agents 能明显降低错误率、提高验证性、减少重复劳动时，优先使用。
2. 选择工具时优先考虑：
   - 是否能直接读取真实上下文
   - 是否能减少猜测
   - 是否能提供可复现证据
3. 若环境可用，优先使用现代命令：
   - 搜索：`rg`、`fd`、`ast-grep` (`sg`)
   - 结构化处理：`jq`、`yq`
   - 差异与定位：`git diff --stat`、`git grep`、`git log -- <path>`
   - 进程与端口：`pgrep`、`pkill`、`lsof`、`ss`
4. 工具输出必须结合仓库上下文解释，不能机械照搬。
5. 对新技术栈问题，若本地证据不足，必须查文档、issues、release notes；**不要反复试过期写法**。

---

## 6. 明确禁止

1. 为了通过 lint / test / build，**绕过问题本身**，包括但不限于：
   - 注释、删除、跳过关键逻辑
   - 吞掉错误或静默降级
   - 伪造返回值、伪造成功状态
   - 用临时 mock 掩盖真实失败
   - 只改表面使检查通过，不解决根因
2. 非必要禁止：
   - 引入新依赖
   - 新增抽象层
   - 修改目录结构
   - 大规模重构
   - 改动基础设施、构建链路、数据库结构、鉴权等关键路径
3. 禁止顺手修复无关问题，除非它阻塞当前任务。
4. 禁止把猜测包装成结论，把未验证包装成完成。

---

## 7. 标准工作流

1. **明确目标**
   - 提炼目标、输入、输出、约束、验收标准、风险
2. **审查现状**
   - 阅读相关代码、配置、文档、测试
   - 理解现有实现路径、边界与依赖
3. **制定方案**
   - 优先最小闭环
   - 若有多个方案，说明取舍与额外复杂度是否值得
4. **实施改动**
   - 保持改动局部化
   - 保持命名、风格、结构与现有约定一致
5. **验证结果**
   - 运行与改动直接相关的测试、构建、检查
   - 无法运行时，说明原因、影响与替代验证方式
6. **输出结果**
   - 改了什么
   - 为什么这样改
   - 验证到什么程度
   - 剩余风险
   - 回滚点或下一步建议

---

## 8. 风险升级条件

遇到以下情况，必须显式提示风险，不得闷头推进：

1. 需求与现有架构明显冲突
2. 改动影响多个核心模块或公共契约
3. 需要新增关键依赖或改动关键链路
4. 测试缺失导致无法可靠验证
5. 现有技术债已使继续叠加改动风险过高
6. 需要依赖外部文档或版本差异才能判断正确实现

---

## 9. 多 Agent 协作

需要多 agent 协作时，按需分工，不强制角色齐全：

- Planner：拆解目标、范围、验收标准、风险
- Architect：边界、接口、数据流、取舍
- Implementer：最小改动落地
- Reviewer：正确性、复杂度、回归风险
- Tester：验证设计与执行
- Docs：更新文档、迁移说明、经验沉淀

原则：

1. 分工必须服务于降低错误率与减少返工。
2. 子 agent 的输出必须回到仓库上下文中校验，不能直接照搬。
3. 若已有合适 skills / MCP / tools，优先调用，而不是徒增 agent 层级。

---

## 10. 经验沉淀

1. 任务中形成的**已验证、可复用**经验，按领域沉淀到 `.agents/playbooks/`。
2. 推荐分类：
   - `.agents/playbooks/debugging/`
   - `.agents/playbooks/tauri/`
   - `.agents/playbooks/axum/`
   - `.agents/playbooks/sveltekit/`
   - `.agents/playbooks/build/`
   - `.agents/playbooks/testing/`
   - `.agents/playbooks/release/`
3. 适合沉淀的内容：
   - 版本兼容性结论
   - 常见报错与根因
   - 调试流程
   - 命令模板
   - 跨模块协作约定
4. 不适合沉淀的内容：
   - 未验证猜测
   - 一次性日志
   - 与单次临时任务强绑定的上下文
5. 若发现现有 playbook 过期、冲突或失真，应在任务结束时同步修订。

---

## 11. 执行底线

1. 不回避问题，不制造表面通过。
2. 不盲猜，不重复试错同一类过期方案。
3. 不为了“架构好看”牺牲交付效率。
4. 不为了短期通过破坏长期可维护性。
5. 始终优先：真实证据、局部改动、可验证结果。
