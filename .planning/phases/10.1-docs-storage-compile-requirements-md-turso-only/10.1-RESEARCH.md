# Phase 10.1 Research — Turso-only 存储持久化、编译裁剪与多租户测试体系

**Date:** 2026-04-08  
**Mode:** Standard research (Level 1.5: known stack + concrete收敛策略确认)

## Scope Understanding

根据 `10.1-CONTEXT.md` 的锁定决策（D-01 ~ D-16），本阶段必须同时完成：

1. 服务端 + 桌面端默认运行路径统一为 Turso 文件库（禁止默认 memory）。
2. 启动失败 fail-fast（路径创建、连接、迁移任一失败都不允许降级）。
3. 默认编译/测试/E2E 仅走 Turso，其他存储 adapter 保留可选 feature。
4. 多租户测试扩展到 `settings/theme/counter/agent` 四维，并保持 fixed tenant pair + fail-fast。

## Current Codebase Findings

### 1) 运行态与存储现状

- `servers/api/src/state.rs` 当前 `new_dev()` 明确使用 `Surreal memory` + `EmbeddedTurso(:memory:)`，与 D-01/D-02 冲突。
- `servers/api/src/main.rs` 当前固定 `AppState::new_dev()`，未走配置化路径。
- `servers/api/src/config.rs` 当前默认 `provider=SurrealDB`、`url=memory`，与 Turso-only 默认冲突。
- `apps/client/native/src-tauri/src/lib.rs` 当前桌面端也用 `EmbeddedTurso::new(":memory:")`，与 D-01/D-02 冲突。

### 2) 编译依赖与 feature 现状

- `servers/api/Cargo.toml` 同时硬依赖 `storage_surrealdb` + `storage_turso`。
- 根 `Cargo.toml` 工作区包含全部 adapter 成员（合理），但默认执行链路未做 feature 收敛。
- `moon.yml` 与 `.github/workflows/e2e-tests.yml` 当前虽已偏向 Turso 路径，但未显式声明默认 lane 禁止触发 surreal/libsql 编译路径。

### 3) 多租户测试现状

- Web 与 Desktop 均已有固定租户 (`tenant_a_user/tenant_b_user`) 和 reset/init helper，可复用 D-13。
- 现有断言主要集中在 counter 隔离；settings/theme/agent 维度还未形成同等强度的可重复回归。

## External Validation (Context7)

使用 Context7 查询 `/tursodatabase/turso`：

- `Builder::new_local("path/to.db")` 是 file-backed 推荐方式。
- `Builder::new_local(":memory:")` 可用于测试/临时用途，适合放在 `#[cfg(test)]` 下。
- 启动阶段连接失败应直接返回错误（符合本阶段 fail-fast）。

这与 D-02 / D-03 完全一致：memory 仅限测试路径，生产/默认路径必须文件库且失败即终止。

## Recommended Implementation Strategy

### A. 先收敛“路径与启动契约”（服务端 + 桌面端）

1. 服务端配置默认改为 Turso + 文件路径。
2. 增加路径解析与目录自动创建逻辑（env 覆盖 + default 来源可识别）。
3. `main.rs` 从 `new_dev()` 切换到配置化 `new_with_config()`，保证默认运行即 Turso-only。
4. 启动日志打印 provider、绝对路径、来源（default/env），并避免敏感值。

### B. 再做 Cargo feature 裁剪（默认 Turso，其他可选）

1. `runtime_server` 对 `storage_surrealdb` 改为 optional feature，默认关闭。
2. 保持 `storage_turso` 为默认链路。
3. 在 moon/CI 默认命令中显式只跑默认 features，避免隐式触发可选 adapter 编译。

### C. 最后扩展测试矩阵（四维隔离）

1. 复用固定 tenant fixture（D-13）。
2. 在 Web/Desktop 增加 settings/theme/agent 的隔离断言（D-14）。
3. 统一 fail-fast：任一 init/reset/assert 失败立即退出（D-16）。
4. 保持可替换认证契约：测试桩入口明确且可被真实认证替换（D-15）。

## Risk Register

1. **高风险：** 一次性改动服务端状态初始化可能影响 `/readyz` 与 E2E 启动时序。  
   - 缓解：先补路径/迁移单元测试，再切入口函数。
2. **中风险：** Cargo feature 收敛可能造成未预期编译缺失。  
   - 缓解：默认链路 + 可选 feature 各自加 `cargo check` 命令验证。
3. **中风险：** 四维隔离测试可能引入 flaky。  
   - 缓解：沿用 serial + known seed reset + retry with bounded limit。

## Validation Architecture

本阶段验证采用“三层闸门”：

1. **配置/路径层**：确认默认 provider/path/source 可观测，且无默认 memory。
2. **编译层**：默认 `cargo check --workspace` 不触发 surreal/libsql adapter 编译链；可选 feature 单独可编译。
3. **行为层**：Web + Desktop 多租户四维隔离可重复通过，并保留可诊断证据。

建议快速验证命令：

- `cargo test -p runtime_server`
- `cargo test -p native-tauri --lib`
- `cargo check -p runtime_server --no-default-features --features storage-surreal`
- `moon run repo:test-e2e`
- `moon run repo:test-desktop`

## Output for Planner

推荐拆分为 4 个 plan：

1. API Turso-only 路径与 fail-fast 启动收敛。
2. Native Tauri Turso 文件库路径与启动可观测性收敛。
3. Cargo/CI 默认编译裁剪（Turso-only default, optional feature opt-in）。
4. 多租户四维隔离测试扩展与 fail-fast 证据化。
