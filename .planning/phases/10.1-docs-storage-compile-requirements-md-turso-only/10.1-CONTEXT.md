**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

依据 `docs/STORAGE_COMPILE_REQUIREMENTS.md`，在不新增业务能力的前提下，完成 Turso-only 存储持久化、默认编译链路裁剪与多租户测试体系增强。

本阶段聚焦于“存储落点、默认编译面、测试可重复性与证据链路”的实现口径，不扩展新产品能力域。

</domain>

<decisions>
## Implementation Decisions

### 存储提供者与运行时持久化口径
- **D-01:** 默认运行面收敛为 Turso-only：desktop 与 `runtime_server` 默认路径均使用 Turso 文件数据库。
- **D-02:** 默认运行路径禁止 `:memory:`/`memory`；内存库仅允许在测试专用路径（如 `#[cfg(test)]` 或测试 fixture）中出现。
- **D-03:** Turso 文件库初始化失败（路径创建、连接、迁移任一失败）统一 fail-fast，进程启动失败，不允许回退 memory 或静默降级。
- **D-04:** 租户域数据（`tenant`/`user_tenant`）与业务域数据（`counter`、`agent conversations/messages`）统一落 Turso，避免跨存储语义分裂。

### 默认编译与 feature 裁剪
- **D-05:** 默认开发/测试/CI 编译链路仅包含 Turso 方案。
- **D-06:** `storage_surreal` 与 `storage_libsql` 保留为可选 feature，默认关闭；仅在显式 feature/job 下参与编译与测试。
- **D-07:** 主二进制与默认任务依赖图不得硬依赖未启用 adapter（满足“未启用模块不参与默认编译”）。
- **D-08:** 默认 E2E 门禁仅验证 Turso 路径，不触发 Surreal/libsql 适配器编译链路。

### 路径策略与可观测性
- **D-09:** Tauri 业务库默认使用平台标准应用数据目录下固定 DB 文件名，并支持环境变量覆盖。
- **D-10:** Axum 业务库默认使用 `servers/api` 下固定 DB 文件，并支持环境变量覆盖。
- **D-11:** 若目标路径父目录不存在，启动时自动创建。
- **D-12:** 启动日志必须打印 provider、最终生效绝对路径、路径来源（default/env）；禁止输出敏感 token/密钥。

### 多租户与用户测试模型
- **D-13:** 延续 Phase 10 已锁定的固定双租户身份（`tenant_a_user`/`tenant_b_user`）与可重复初始化策略。
- **D-14:** 本阶段回归基线扩展为四类隔离维度：`settings`、`theme`、`counter`、`agent`（tenant + user 维度）。
- **D-15:** 认证场景采用“邮箱/密码测试桩契约可替换”策略：本阶段允许受控测试通道，但契约必须可替换为真实认证实现。
- **D-16:** 多租户测试统一 fail-fast：任一 tenant/user 初始化或隔离断言失败即失败，禁止“继续跑完再汇总”的弱门禁策略。

### the agent's Discretion
- 具体 feature 名称与组合方式（例如 `storage-turso`、`storage-surreal`、`storage-libsql` 的声明落点与连线细节）。
- 具体环境变量命名（桌面路径/服务端路径覆盖变量）与默认 DB 文件名。
- 迁移脚本组织方式、目录布局与执行入口（在满足 D-03/D-04/D-11 前提下）。
- 四类隔离回归的最小用例拆分与断言实现细节（在满足 D-14/D-16 前提下）。

</decisions>

<specifics>
## Specific Ideas

- “默认仅 Turso”与“非 Turso adapter 可选 feature 默认关闭”是本阶段硬约束。
- “彻底 Turso-only”包含 tenant 域与业务域统一落库，不允许只迁业务不迁租户域。
- 路径策略必须做到“有默认值 + 可覆盖 + 可观测 + 可自动建目录”，不接受仅靠人工约定。
- 测试模型不仅是 counter 隔离，必须覆盖 `settings/theme/counter/agent` 四类维度，并统一 fail-fast。

</specifics>
