## 1. JWT Signature Verification

### Current State

**File:** `servers/api/src/middleware/tenant.rs:8`

```rust
use jsonwebtoken::dangerous::insecure_decode;
```

- Uses `dangerous::insecure_decode` — payload-only, NO signature verification
- No `exp` claim checking — expired tokens accepted indefinitely
- Tests use `insecure_decode` too (needs updating)

### Problem

Any attacker can forge arbitrary JWT tokens. The `sub` claim becomes the tenant_id, so a forged token grants access to any tenant.

### Solution

**Config already has JWT secret:** `servers/api/src/config.rs` → `AuthConfig { jwt_secret, jwt_expiration_secs }`

**Approach:** Environment-gated switch using `jsonwebtoken::decode()` + `Validation` + `DecodingKey::from_secret()`.

```rust
use jsonwebtoken::{decode, DecodingKey, Validation, Algorithm};

// Gate: if APP_AUTH__JWT_SECRET is set to "dev-secret-change-in-production",
// fall back to insecure mode (dev only). Otherwise, verify signature + exp.
let mut validation = Validation::new(Algorithm::HS256);
validation.validate_exp = true;

let token_data = decode::<Claims>(
    &token,
    &DecodingKey::from_secret(jwt_secret.as_ref()),
    &validation
)?;
```

**Key decisions:**
- Keep HS256 (symmetric) — simplest for boilerplate, no JWKS needed
- Dev mode fallback: if `jwt_secret` == `"dev-secret-change-in-production"`, use `insecure_decode` (with warning log)
- Prod mode: full signature + exp validation
- Tests: use `EncodingKey::from_secret` + `decode` (both sides of same key)

**Error handling from jsonwebtoken docs:**
- `ErrorKind::InvalidSignature` → 401
- `ErrorKind::ExpiredSignature` → 401
- `ErrorKind::InvalidToken` → 401

### Test Strategy

- Unit tests with HS256 encode/decode round-trip
- Test expired token rejection
- Test invalid signature rejection
- Test dev-mode fallback still works

## 2. CORS Configuration

### Current State

**File:** `servers/api/src/lib.rs:89`

```rust
.layer(CorsLayer::permissive())
```

### Problem

`CorsLayer::permissive()` allows ALL origins, methods, and headers. In production, any website can make authenticated API calls.

### Solution

Use `CorsLayer::new()` with explicit allow list from environment config.

```rust
use tower_http::cors::{CorsLayer, AllowOrigin};
use axum::http::{header, Method};

let cors = if is_dev_mode {
    CorsLayer::permissive()  // Dev: keep permissive for DX
} else {
    CorsLayer::new()
        .allow_origin(AllowOrigin::whitelist(allowed_origins))
        .allow_methods([Method::GET, Method::POST, Method::PUT, Method::DELETE])
        .allow_headers([header::AUTHORIZATION, header::CONTENT_TYPE, header::ACCEPT])
        .allow_credentials(true)
};
```

**Config approach:** Add `cors_allowed_origins` field to `ServerConfig` in `servers/api/src/config.rs`. Read from `APP_SERVER__CORS_ALLOWED_ORIGINS` env var (comma-separated list).

**Dev mode detection:** If `cors_allowed_origins` is empty, fall back to permissive (dev default). If set, enforce allowlist.

### Test Strategy

- Integration test: request from allowed origin → 200
- Integration test: request from disallowed origin → CORS rejection
- Config test: empty origins → permissive, set origins → enforced

## 3. Sensitive Field Redaction

### Current State

**File:** `packages/contracts/api/src/lib.rs:65-72`

```rust
pub struct AgentConfig {
    pub api_key: String,
    pub base_url: String,
    pub model: String,
}
```

- `api_key` is fully serializable — will appear in JSON responses, logs, debug output
- Both `Serialize` and `Deserialize` derived

### Problem

If `AgentConfig` is ever included in a response or logged (via `tracing` or `Debug`), the API key is exposed.

### Solution

Add `#[serde(skip_serializing)]` on `api_key`. Keep deserialization so the field can still be received in requests. Add a separate non-sensitive display type.

```rust
pub struct AgentConfig {
    #[serde(skip_serializing)]
    pub api_key: String,
    pub base_url: String,
    pub model: String,
}
```

**ts-rs impact:** `#[ts(skip)]` on api_key to avoid TypeScript type having a serializable api_key field.

**Consider:** Also check if `AgentConfig` derives `Debug` — if so, redact api_key in Debug output using a wrapper or manual Debug impl.

### Test Strategy

- Serialize AgentConfig → api_key NOT present in JSON output
- Deserialize AgentConfig → api_key correctly populated
- Debug output → api_key redacted or not shown
