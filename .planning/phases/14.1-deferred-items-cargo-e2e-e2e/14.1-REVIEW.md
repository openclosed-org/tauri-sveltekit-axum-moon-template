# Phase 14.1: Code Review Report

## Summary

本次审查覆盖了 14.1-01/02/03 涉及的测试夹具、E2E 编排脚本、CI 工作流以及一个相关影响文件（desktop tenant isolation）。

整体方向正确：鉴权契约对齐、runtime preflight、lane 摘要都提升了可诊断性。但存在几个会影响测试可靠性与可维护性的点，主要集中在：
- CI web lane 与 runtime 启动依赖不一致（可能导致超时失败）
- 基于 `process.cwd()` 的 workspace 路径推导脆弱
- E2E 用例中“登录页即提前返回通过”的弱断言，存在假绿风险

## Warnings

### WR-01: CI web-e2e 缺失 Rust toolchain，可能导致 globalSetup 卡超时

**File:** `.github/workflows/e2e-tests.yml:39-64`  
**Issue:** `globalSetup -> ensureWebE2EPreflight -> ensureApiReady` 在 API 未就绪时会尝试 `cargo run -p runtime_server`（`apps/client/web/app/tests/fixtures/runtime.ts:69-73`）。但 `web-e2e` job 只安装了 Node/Bun，没有安装 Rust，导致 `cargo` 不可用时只能轮询到超时并失败。  
**Fix:** 在 `web-e2e` job 增加 Rust setup（至少可执行 `cargo`），或在 job 内显式启动 runtime server 并等待 `/readyz` 后再跑 Playwright。

```yaml
- name: Setup Rust
  uses: dtolnay/rust-toolchain@stable
  with:
    toolchain: 1.94.1

# 可选：显式预热 runtime
- name: Start runtime server
  run: cargo run -p runtime_server &
```

### WR-02: runtime fixture 依赖当前工作目录推导 workspaceRoot，路径易漂移

**File:** `apps/client/web/app/tests/fixtures/runtime.ts:11`  
**Issue:** `path.resolve(process.cwd(), '..', '..', '..', '..')` 依赖调用方 cwd 恰好是 `apps/client/web/app`。一旦在其他入口（不同 cwd、IDE/脚本直调）调用 `ensureApiReady/ensureWebE2EPreflight`，会推导出错误根目录，导致 binary/types 检查误判。  
**Fix:** 改为基于 `import.meta.url` 的文件相对路径推导，避免 cwd 假设。

```ts
import { fileURLToPath } from 'node:url';

const __dirname = fileURLToPath(new URL('.', import.meta.url));
const workspaceRoot = path.resolve(__dirname, '..', '..', '..', '..', '..', '..');
```

### WR-03: counter E2E 多处“重定向到 /login 即返回通过”，存在假绿风险

**File:** `apps/client/web/app/tests/e2e/counter.test.ts:22-27,35-38,47-49,62-64,77-79,96-98,121-126`  
**Issue:** 多个测试在检测到 `/login` 后直接 `return`（判定“guard 正常”），导致计数器核心行为（增减/重置/持久化）在未登录场景下被跳过且仍通过。此模式会掩盖真实回归。  
**Fix:** 将“鉴权守卫验证”和“计数器行为验证”拆分；行为测试应先强约束登录成功（失败即测试失败），而不是 early return。

```ts
await page.goto('/counter');
await expect(page).not.toHaveURL(/\/login/); // 行为测试前置条件
```

**File:** `e2e-desktop-playwright/tests/specs/tenant-isolation.spec.ts:99-160`  
**Issue:** desktop spec 内部重复实现了 `ensureApiReady`、`makeTenantToken`、auth headers，与 web fixtures (`apps/client/web/app/tests/fixtures/auth.ts`, `runtime.ts`) 语义相近但未共享，后续改动易出现一侧修复、另一侧滞后。  
**Fix:** 可考虑抽到共享 test helper（例如 `scripts/e2e` 或 `packages/testing`）并复用，至少统一 token/header 与 runtime readiness 的构造路径。
