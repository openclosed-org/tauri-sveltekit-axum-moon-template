  <name>Task 1: 创建 create-feature playbook（高层步骤模式）</name>
  <files>.agents/playbooks/create-feature.md</files>
  <action>
创建 .agents/playbooks/create-feature.md，采用高层步骤模式（3-5 步概览），适用于日常功能开发。

文件结构必须包含以下章节：

1. **Trigger（触发条件）**：当需要新增功能模块时触发（如新增 API 端点、新增 UI 页面、新增 Rust feature crate）

2. **Pre-flight（前置检查）**：
   - 确认 AGENTS.md 已阅读
   - 确认 .agents/rubrics/boundary-compliance.md 已阅读
   - 运行 `just verify` 确认当前状态干净
   - 确认 feature 归属正确的层（domain/usecases/adapters/hosts/servers）

3. **Execution Steps（执行步骤）** — 5 步高层流程：
   - Step 1: 定义 contracts（如在 packages/contracts/api/ 中新增 DTO）
   - Step 2: 实现 domain port trait（如需要新能力）
   - Step 3: 实现 usecases service trait + impl
   - Step 4: 实现 adapter（storage 或 host）
   - Step 5: 实现 host 入口（Tauri command 或 Axum route）+ 前端组件

4. **Verification（验证步骤）**：
   - 运行 `just typegen` 确保类型生成无漂移
   - 运行 `just verify` 确保所有检查通过
   - 运行 `cargo test` 确保 Rust 测试通过
   - 运行边界检查：确认无跨层违规 import

5. **Rollback（回滚路径）**：
   - `git stash` 或 `git reset --hard HEAD` 回退未提交改动
   - 如已提交：`git revert HEAD` 回退最后一个 commit
   - 清理 generated/ 目录：`rm -rf frontend/generated/` 后重新 `just typegen`

参考 .agents/rubrics/boundary-compliance.md 的格式风格（Markdown 标题 + checkbox 列表）。

<task type="auto">
  <name>Task 2: 创建 update-contracts playbook（详细步骤模式）</name>
  <files>.agents/playbooks/update-contracts.md</files>
  <action>
创建 .agents/playbooks/update-contracts.md，采用详细步骤模式，适用于 contracts 变更这一关键路径。

文件结构必须包含以下章节：

1. **Trigger（触发条件）**：当需要修改跨边界 DTO 时触发（如新增/修改 API 请求响应结构、修改 Tauri 命令参数、修改事件 payload）

2. **Pre-flight（前置检查）**：
   - 确认 AGENTS.md 已阅读
   - 确认 .agents/rubrics/boundary-compliance.md 已阅读
   - 运行 `just verify` 确认当前状态干净
   - 确认 contracts 修改范围（packages/contracts/api/ 还是 auth/ 还是 events/）
   - 创建 git branch 隔离改动

3. **Execution Steps（执行步骤）** — 详细 7 步流程：
   - Step 1: 在 contracts crate 中修改 DTO 结构体（添加/删除/修改字段）
   - Step 2: 确保结构体同时标注 #[derive(TS, utoipa::ToSchema)] 和 serde 属性
   - Step 3: 运行 `just typegen` 生成 TypeScript 类型到 frontend/generated/
   - Step 4: 更新 Rust 侧消费者（servers/api 路由、runtime_tauri 命令 handler）
   - Step 5: 更新 TypeScript 侧消费者（SvelteKit routes、Svelte components）
   - Step 6: 运行 `just typegen` 再次确认无新漂移
   - Step 7: 运行 `just verify` 全量检查

4. **Verification（验证步骤）** — 必须包含以下命令：
   - `just typegen` — 确认类型生成成功且无未提交 diff
   - `just verify` — 确认全量检查通过
   - `cargo test` — 确认 Rust 测试通过
   - 检查 frontend/generated/ 文件已 git add（不漂移）
   - 边界检查：确认 contracts 未引入 domain/usecases import

5. **Rollback（回滚路径）**：
   - 未提交：`git checkout -- packages/contracts/` 丢弃 contracts 改动
   - 未提交：`rm -rf frontend/generated/` 清理生成的类型
   - 已提交：`git revert HEAD` 回退
   - 如 typegen 失败：检查 contracts crate 编译错误，修复后重试

6. **Drift Prevention（漂移防护）**：
   - 解释为什么 contracts 是单一真理源
   - 说明 CI 中 drift check 的工作原理
   - 列出常见漂移场景及修复方法

参考 .agents/rubrics/boundary-compliance.md 的格式风格。
