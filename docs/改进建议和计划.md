📋 改进建议和计划
一、架构层面（高优先级）

1. 修复已知依赖违规

- 问题：feature-auth 违反了架构规则，依赖了 usecases
- 位置：packages/features/auth/Cargo.toml
- 影响：破坏了依赖反转原则，降低了模块独立性
- 建议：将 usecases 依赖移到 adapters 层，feature-auth 只定义 trait

2. 解决类型漂移问题

- 问题：DashboardStats 类型在 feature-admin 和 contracts/api 中存在 i64 vs u64 不一致
- 位置：packages/features/admin/ vs packages/contracts/api/
- 影响：跨层类型不一致，可能导致运行时错误
- 建议：统一到 contracts/ 作为唯一真理源

3. 消除类型重复

- 问题：UserProfile 在 feature-auth 和 contracts/auth 中重复定义
- 位置：packages/features/auth/ vs packages/contracts/auth/
- 建议：删除 feature-auth 中的重复定义，统一引用 contracts/auth

---

二、安全层面（高优先级）

1. JWT 签名验证缺失

- 问题：当前使用 dangerous::insecure_decode 进行 JWT 解码，没有验证签名
- 位置：servers/api/src/middleware/tenant.rs:8
- 风险：攻击者可以伪造任意 JWT token
- 建议：
  - 短期：添加环境变量开关，生产环境强制启用签名验证
  - 中期：实现 JWKS（JSON Web Key Set）验证
  - 长期：支持多种 OAuth 提供商的 JWKS

2. JWT 过期时间未检查

- 问题：insecure_decode 不检查 exp 声明
- 位置：servers/api/src/middleware/tenant.rs:35
- 风险：过期的 token 仍然可以使用
- 建议：在 v2 中添加过期时间检查

3. CORS 配置过于宽松

- 问题：当前使用 CorsLayer::permissive()，允许所有来源
- 位置：servers/api/src/lib.rs:89
- 风险：生产环境可能被恶意网站利用
- 建议：生产环境配置明确的允许列表

4. 敏感信息泄露风险

- 问题：AgentConfig 包含 api_key 字段，可能通过日志泄露
- 位置：packages/contracts/api/src/lib.rs:66-72
- 建议：
  - 使用 #[serde(skip_serializing)] 或 #[ts(skip)] 避免序列化
  - 添加敏感字段标记
