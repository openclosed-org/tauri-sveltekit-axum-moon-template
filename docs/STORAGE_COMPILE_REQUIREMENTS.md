# 存储与编译策略开发需求文档（Turso-only 阶段）

## 1. 文档目标

本需求文档用于指导后续 agent 对当前项目进行存储重构、编译裁剪、测试增强。

目标是：

1. 明确不同数据必须存到哪里。
2. 在当前阶段只启用 Turso 存储方案。
3. 未启用的存储模块不参与默认编译、测试、E2E。
4. 建立可验证的多租户测试体系（2 tenant + 对应用户）。

---

## 2. 阶段范围（In Scope）

### 2.1 本阶段必须完成

1. 桌面端业务数据改为本地 Turso 文件数据库持久化（禁止 `:memory:`）。
2. Axum 后端业务数据改为 Turso 文件数据库持久化（禁止 `:memory:`）。
3. 默认编译链路仅包含 Turso 方案。
4. libsql 与 surrealdb 暂不进入默认编译、默认测试、默认 E2E。
5. 多租户与多用户测试数据体系可稳定复现。

### 2.2 本阶段不做

1. 不启用 SurrealDB 作为默认后端。
2. 不启用 libsql 作为默认后端。
3. 不做离线同步功能最终定版（仅保留后续扩展位）。

---

## 3. 数据分类与存储归属（强约束）

### 3.1 客户端配置数据（轻量配置）

1. `settings`（如 API key/base_url/model）
   - 存储介质：Tauri Store（JSON 文件）。
   - 备注：可继续沿用现有 `settings.json`。

2. `theme-preference`
   - 存储介质：`localStorage`。
   - 备注：保持前端现有行为，不迁移到 Turso。

3. `auth/session`（token、过期时间、用户资料）
   - 存储介质：Tauri Store（JSON 文件）。
   - 备注：保持当前 `auth.json` 机制。

### 3.2 客户端业务数据（结构化、可查询）

1. `counter`、`agent conversation/messages`、后续业务实体
   - 存储介质：本地 Turso（SQLite 文件，非内存）。
   - 强约束：禁止 `:memory:`。

### 3.3 服务端业务数据（Axum）

1. 租户域数据（tenant、user_tenant）及业务数据
   - 存储介质：Axum 进程对应 Turso 文件数据库。
   - 强约束：禁止 `memory` / `:memory:` 作为默认运行模式。

2. 缓存数据
   - 存储介质：可保留内存缓存（如 moka）。
   - 说明：缓存不等于持久化数据库。

---

## 4. 路径策略（当前阶段）

### 4.1 Tauri 本地业务库路径

1. 默认路径要求：与桌面应用运行目录同级/同目录（按产品需求实现）。
2. 路径需要可配置（后续可暴露设置项，当前可先环境变量或配置文件）。
3. 路径不存在时自动创建父目录。

### 4.2 Axum 本地业务库路径

1. 默认路径要求：`servers/api` 应用根目录下固定 db 文件。
2. 路径支持通过环境变量覆盖。
3. 启动时必须打印最终生效路径（便于排查）。

---

## 5. 多租户与用户测试模型（必须落地）

### 5.1 基础测试身份

至少准备以下测试主体：

1. Tenant A + User A
2. Tenant B + User B

每个用户在测试中需要具备独立身份数据，且跨 tenant 严格隔离。

### 5.2 认证场景覆盖

1. 邮箱登录（测试账号）
2. 密码登录（测试账号）

说明：若当前代码尚未具备完整邮箱密码登录能力，先以测试桩/受控测试通道建模，但最终验收必须具备可替换为真实认证实现的契约。

### 5.3 必测隔离维度

1. settings 隔离（Tenant A/B、User A/B 互不污染）。
2. theme 隔离（按用户维度验证前端本地状态）。
3. counter 隔离（tenant 维度）。
4. agent 配置与会话数据隔离（tenant + user 维度）。

---

## 6. 数据库与表结构要求

### 6.1 原则

1. 所有业务表必须显式定义主键、租户键、必要索引。
2. 与租户相关的表必须包含 `tenant_id`。
3. 与用户强关联的数据必须包含 `user_id`/`user_sub`（按现有域模型）。
4. 必须有迁移机制（幂等，支持重复执行）。

### 6.2 最低表结构检查清单

1. tenant / user_tenant
2. counter
3. agent conversations
4. agent messages
5. 必要唯一约束与查询索引

---

## 7. 编译与依赖裁剪要求（核心）

### 7.1 目标

默认开发、测试、E2E 只走 Turso 方案，避免未启用模块拖慢编译或引入失败。

### 7.2 必须满足

1. `runtime_server` 默认不引入 SurrealDB 与 libsql 存储实现。
2. `native-tauri` 默认只引入 Turso 所需存储依赖。
3. libsql 与 surrealdb 改为可选 feature（非默认）。
4. CI 默认流水线不运行 libsql/surrealdb 对应测试集。
5. E2E 默认只验证 Turso 路径。

### 7.3 feature 策略（建议）

1. `storage-turso`（默认开启）
2. `storage-surreal`（默认关闭）
3. `storage-libsql`（默认关闭）

并要求：任何 storage adapter 未启用时，不得被主二进制强依赖引用。

---

## 8. 安全与数据要求

1. 测试账号与测试数据必须隔离于生产配置。
2. 本地敏感数据（如 token、密钥）不得写入日志。
3. 数据库路径与文件权限应遵循最小权限原则。
4. 禁止通过“跳过逻辑/吞错”方式通过测试。

---

## 9. 验收标准（Definition of Done）

### 9.1 功能验收

1. Tauri 启动后，业务数据持久化到文件数据库，重启不丢失。
2. Axum 启动后，业务数据持久化到文件数据库，重启不丢失。
3. 不存在默认 `:memory:` 路径。

### 9.2 多租户验收

1. Tenant A/B 的 counter、agent、settings 数据互不污染。
2. User A/B 的登录与数据读写互不污染。

### 9.3 编译验收

1. 默认命令仅编译 Turso 所需依赖。
2. 默认测试/E2E 不触发 libsql/surrealdb 编译链路。
3. 切换 feature 后可启用其他 adapter（保留后续扩展性）。
